# Как работают хуки

Перед тем как писать хуки — важно понять что происходит под капотом. Этот раздел не про код, а про концепцию.

> Документация написана так, чтобы читать её линейно сверху вниз — каждый раздел опирается на предыдущий. Но если ты уже знаешь основы и ищешь конкретную тему — можешь переходить напрямую по ссылкам.

---

## Что такое хук

exteraGram — это обычное Android-приложение написанное на Java. Оно постоянно что-то делает: отправляет запросы на серверы Telegram, получает ответы, рисует интерфейс, реагирует на нажатия. Всё это — вызовы функций внутри приложения.

**Хук** — это перехватчик. Ты говоришь приложению: «когда будешь вызывать вот эту функцию — сначала вызови меня». Приложение вызывает твой код, ты делаешь что нужно, и потом либо разрешаешь исходному действию произойти, либо отменяешь его, либо меняешь что-то в нём.

Представь это как таможню: каждый «груз» (запрос, сообщение, вызов метода) проходит через твой контрольный пункт. Ты можешь пропустить, досмотреть и изменить, или развернуть обратно.

---

## Как это технически работает

Плагины в exteraGram запускаются через **Chaquopy** — это Java-библиотека которая позволяет исполнять Python-код внутри Android-приложения. Твой `.plugin` файл — это просто Python, который работает прямо внутри процесса Telegram.

Для перехвата Java-методов используется механизм **Xposed** (конкретно — порт от Aliucord). Это означает что можно перехватить вызов *любого* Java-метода внутри приложения: до его выполнения, после, или полностью заменить.

Тебе не нужно знать детали ни Chaquopy ни Xposed — это детали реализации. Важно понимать что происходит с точки зрения твоего кода.

## Три вида хуков в Reunion

В Reunion есть три вида хуков — они отличаются тем *что именно* ты перехватываешь.

### 1. Хуки на сетевые запросы

Telegram общается с серверами через протокол MTProto. Каждое действие — это запрос определённого типа. Напечатал текст — уходит `TL_messages_setTyping`. Отправил сообщение — `TL_messages_sendMessage`. Вышел онлайн — `TL_account_updateStatus`.

Ты можешь перехватить любой из этих запросов **до отправки** (`pre_request`) или **после получения ответа** (`post_request`).

```
plugin GhostMode {
    fun on_load() {
        hook "TL_messages_setTyping"      // регистрируем интерес
        hook "TL_account_updateStatus"
    }

    fun pre_request(request_name, account, request) {
        // вызывается перед каждым из зарегистрированных запросов
    }
}
```

Подробнее — в разделе [Хуки на сетевые запросы](https://github.com/shareui/reunion-compiler/tree/main/docs/06_network_hooks.md).

### 2. Хук на отправку сообщения

Это специальный хук который срабатывает в момент когда пользователь нажимает «Отправить» — ещё до того как сообщение ушло на сервер. Ты получаешь объект `params` с текстом, можешь его изменить или отменить отправку.

```
plugin WeatherBot {
    fun on_load() {
        hook_send_message
    }

    fun on_send_message(account, params) {
        // params.message — текст который пользователь собирается отправить
    }
}
```

Подробнее — в разделе [Хуки на отправку сообщений](https://github.com/shareui/reunion-compiler/tree/main/docs/07_message_hooks.md).

### 3. Хуки на Java-методы (Xposed)

Самый мощный вид. Можно перехватить вызов *любого* метода внутри приложения — не только сетевые запросы, а вообще любой. Например, метод который рисует заголовок экрана, или метод который проверяет подписку, или конструктор UI-компонента.

```
method_hook TitleLogger {
    fun before(param) {
        val title = param.args[0]
        param.args[0] = "[Hooked] $title"
    }
}
```

Подробнее — в разделе [Хуки на Java-методы](https://github.com/shareui/reunion-compiler/tree/main/docs/08_java_hooks.md).

## Терминаторы — обязательное завершение

Это концепция общая для всех трёх видов хуков. Каждый обработчик должен явно сказать что делать дальше. В сетевых хуках и хуке сообщений есть три варианта:

| Терминатор | Что происходит |
|---|---|
| `default` | Всё идёт как шло — запрос уходит / сообщение отправляется без изменений |
| `modify request` / `modify params` | Запрос / сообщение отправляется, но с твоими изменениями |
| `cancel` | Запрос / сообщение **не отправляется** вообще |

Компилятор проверяет что **каждая ветка** выполнения заканчивается терминатором. Если забудешь — получишь `ReunionSemanticError` при компиляции. Автоматически добавляется `default` только если ни одного терминатора нет вообще и это единственная ветка.

## Порядок выполнения

Важно понимать что происходит в какой момент:

```
Пользователь нажимает "Отправить"
        ↓
on_send_message — твой код
        ↓ (если modify params или default)
Сообщение идёт на сервер как TL-запрос
        ↓
pre_request — твой код (если хукнул этот запрос)
        ↓ (если modify request или default)
Запрос уходит на серверы Telegram
        ↓
Ответ приходит
        ↓
post_request — твой код
        ↓
Приложение обрабатывает ответ
```

Xposed-хуки работают на уровне Java-методов — они могут срабатывать на любом из этих этапов, и не только.
